SETUP
-----
1. Clone the repository:
      git clone <repo-url>
      cd Disney-SD-Smart-Vertical-Streaming

2. Create the virtual environment:
      python -m venv sd_disney

3. Activate it:
      Windows (PowerShell):  .\sd_disney\Scripts\Activate.ps1
      Windows (CMD):         sd_disney\Scripts\activate.bat
      Mac/Linux:             source sd_disney/bin/activate

4. Install dependencies:
      pip install -r requirements.txt


RUNNING THE UI
--------------
From the project root, double-click or run:
      .\run_ui.bat

This launches StreamJup.py via Streamlit without needing to cd into the UI folder.

If the .bat launchers are broken (stale venv path), bypass them with:
      & ".\sd_disney\Scripts\python.exe" -m streamlit run UI\StreamJup.py

Smart Vertical Basketball Streaming - Session Summary
======================================================

PROJECT OVERVIEW
----------------
Disney Senior Design Project: Smart Vertical Basketball Streaming
Primary goal: Automatically reframe landscape basketball video to 9:16 vertical format using YOLO detection + built-in ByteTrack tracking (via ultralytics).

FILES MODIFIED / CREATED
-------------------------
- UI/StreamJup.py                   -> Main Streamlit UI (active file)
- run_ui.bat                        -> Launches StreamJup.py via sd_disney virtual environment
- testing/ball_framing_integration.py -> Diagnosed but not modified (ball detection issue)

FEATURES IMPLEMENTED IN StreamJup.py
--------------------------------------
1. Video upload via Streamlit file uploader
2. YOLO person detection (yolov8m.pt, class 0, conf=0.50, iou=0.95)
3. Frame Scrubber tab - slider to browse frames with detection overlay
4. Play Video tab:
   - Renders clean frames to H.264 (avc1) MP4 via OpenCV VideoWriter
   - Serves video over a local HTTP server (avoids Streamlit 200MB websocket limit)
   - HTML5 <video> + <canvas> overlay with requestAnimationFrame loop
   - Per-frame bounding box data embedded as JSON in the HTML
   - Toggle button to show/hide boxes during playback (no page reload)
5. Metrics: total frames, avg people/frame, peak people

BUGS FIXED DURING SESSION
--------------------------
1.  Play Video button not visible        -> Switched from radio buttons to st.tabs
2.  Video not playing (black player)     -> Switched from mp4v to avc1 (H.264) codec
3.  FileNotFoundError for ffmpeg         -> Removed subprocess; avc1 works natively on Windows 10
4.  use_container_width deprecation      -> Replaced with width='stretch' throughout
5.  Websocket 200MB limit exceeded       -> Replaced base64 embedding with local HTTP server
6.  Video letterboxed / boxes misaligned -> Changed CSS from object-fit:contain to width:100%; height:auto
7.  Toggle button cut off at bottom      -> Moved controls div above player-wrap
8.  iframe too short (video cut off)     -> Changed container width estimate from 900px to 1400px
9.  OpenH264 terminal warnings           -> Added OPENCV_LOG_LEVEL and OPENCV_FFMPEG_LOGLEVEL env vars
10. Temp video files accumulating on disk (OS temp dir filling up):
    - "Render & Play" re-encoded the full video to a new temp file on every click
    - Player and ball video servers shared a single _active_server dict, so starting
      one would delete the other's temp file (causing a broken player on re-open)
    - No cleanup occurred on new video upload or app exit
    Fix: cache rendered file path in session_state and reuse it on repeat clicks;
    split _active_server into _active_servers keyed by label ("player"/"ball");
    delete old rendered files on new upload; atexit handler cleans up all temp
    files when the process exits.

KNOWN / PENDING ISSUES

  If still cut off, increase estimate to 1600px or implement dynamic height via JS postMessage.
